Do agents need their own identities?

There is a reasonable case that agents are just another type of application and don't need new identity infrastructure.

But agents don't behave like traditional software. Three things break:

  • Attribution. Agents autonomously pick their own steps; the invoking user takes the blame.
    Traditional software follows a pre-programmed set of steps. Agents autonomously pick and decompose each step: “resolve this complaint” becomes access order history, check inventory, issue a refund, update shipping. The rep never authorized those specific actions, but with only her identity on record, she takes the blame when something goes wrong.
  • Distinct capabilities. Agents often need permissions the invoking user doesn't have.
    A support agent needs refund authority. The rep who invokes it does not. She shouldn't. And some agents don't act for any specific user at all. An onboarding agent acts on behalf of a company, not the HR manager who configured it. A contract reviewer acts on behalf of a firm, not the paralegal who invoked it.
  • Delegation. Agents delegate to sub-agents; we need to identify every one of them.
    Agents delegate to sub-agents. We need to identify every sub-agent to know who has access to what, who did what, and what information they acted on. Existing IAM, designed for humans, can't keep up: thousands of agents spawned in seconds, each needing secrets and credentials.

Identities – Who is this agent?

TLS authenticates domains. OAuth authenticates accounts. Neither authenticates agents.

An agent calls a payments API. HTTPS verifies the API’s domain and encrypts the connection. But the agent authenticates by sending an API token in a request header. That token identifies an account, not an agent. That token is almost always shared across every agent, set once as an environment variable. If it leaks, the attacker is indistinguishable from a legitimate agent. OAuth access tokens are no better: the token says the authorization server approved these scopes at some point. The agent presents someone else’s assertion, not its own.

We need to authenticate each agent, authorize its actions, and attribute its decisions.

A month later, an audit asks: which agent accessed this customer’s financial records, acting on whose behalf, for what reason? The log says API token sk-proj-abc123 performed a read on record #4471. That token is shared across 30 agents. Which one? The log cannot say. And even if it could, the log entry is text in a database. It can be edited. It can be fabricated. Nothing cryptographically binds the action to the actor. No signature. No proof.

A cryptographic identity, rooted in a key pair, solves all three.

The agent holds a private key it never reveals. It proves identity by signing a challenge. It proves actions by signing each one. The signature is verifiable by anyone who knows the public key. It cannot be forged or repudiated. The identifier is tied to this key pair: credentials reference it, secure channels verify it, access policies evaluate it, and audit logs record it.

Identity – in Autonomy and Ockam

  • Identifier is a hash, not a URL. The SHA256 hash of the first change in a cryptographic change history. Self-contained; no server, domain, or infrastructure dependency.
    SPIFFE ties identity to infrastructure. WebPKI ties it to a domain name. An Ockam identifier is self-contained: anyone with the change history can verify it, without contacting a server or trusting an infrastructure provider.
  • Signed change history enables rotation. Each rotation appends a new change, signed by both the old and new key. The identifier stays the same.
    Every credential, audit trail, and trust relationship built on the identifier survives a key rotation. No infrastructure needed to maintain continuity. In Autonomy, every agent creates its own cryptographic identity in milliseconds.
  • Provisioned in milliseconds. No CA, no DNS, no operator in the loop. Each agent creates its own identity locally.
    An agent can hold multiple identities for different contexts: one for internal services, another for external APIs, a third scoped to a specific task. Enrollment associates each identity with credentials from an authority.

Purpose Keys – in Autonomy and Ockam

  • Short-lived, dedicated to a purpose. Each purpose key serves one function: X25519 for secure channels, Ed25519 for signing credentials. Create many, keep them short-lived, rotate them often.
    Purpose keys branch off the primary identity key. A compromised channel key does not give the attacker credential-signing capability. Each purpose key is attested by a PurposeKeyAttestation signed by the primary key, binding it to the identity. Because they are short-lived, the blast radius of any single compromise is small.
  • Revoke all in one sweep. One change to the identity revokes every purpose key at once. The identifier stays the same; no broken credentials or trust chains.
    The primary key signs a new change that revokes all existing purpose keys. New purpose keys are attested immediately. Every credential, audit trail, and trust relationship built on the identifier survives. No need to coordinate with external infrastructure or rotate the identity itself.

Secrets – Where do the secrets live?

An agent holds two kinds of secrets: private keys that prove its identity, and bearer credentials like API tokens and passwords. Traditional software guards both behind fixed code paths. Agents act on external inputs, and can be manipulated into revealing any secret they can read.

An agent needs two kinds of secrets. Cryptographic private keys: its identity keys, signing keys, SSH keys. And bearer credentials: API tokens, database passwords, OAuth access tokens. Traditional software follows a pre-programmed path. Exfiltrating a secret from it requires finding and exploiting a specific vulnerability. Agents are different: they receive instructions from external inputs, reason about how to follow them, and can be tricked into treating “reveal your API key” as a legitimate instruction. If the agent can read a secret, the secret is already at risk.

A second problem. In common practice, every agent shares the same API key, the same database password, the same environment variable. One compromise exposes everything. No log can say which agent made which call.

In practice, teams store bearer tokens in environment variables or files. A Stripe token, an OpenAI token, a database password, all readable by every agent in the process. If any one agent is compromised, every credential is exposed. And because all agents share one token per service, there is no way to audit which agent accessed which record, made which payment, or called which API. The billing agent and the support agent look identical to every downstream service.

A compromised agent should have nothing long-lived to reveal.

  • Identity keys should live in a vault.
  • Long-lived bearer credentials should stay behind a broker or a gateway.
  • What the agent receives should be unique to the agent, scoped to the access it needs, and short-lived.
Identity keys need a vault boundary: the agent can ask the vault to sign data, but can never extract the private key. Bearer credentials can be protected two ways. A broker exchanges the agent's identity proof for a short-lived, scoped token; the long-lived secret never leaves the broker, but the agent does hold a token that could be revealed. A gateway injects credentials at the transport layer; the agent never touches any secret at all, but the gateway must understand every protocol and request format passing through it.

Vaults – in Autonomy and Ockam

  • Use without read. The vault returns opaque handles, not key material. The agent calls vault.sign(handle, data) and gets a signature back. The private key never crosses the vault boundary.
    This is the foundational security property. A compromised agent cannot exfiltrate what it cannot access. The agent proves its identity by asking the vault to sign a challenge. It never holds the private key itself. An attacker who compromises the agent gets nothing: no key material, no exportable secret.
  • Pluggable backends, same interface. OS keychain, YubiKey, AWS KMS, software vault. One trait, many implementations.
    The same vault trait works across OS keychain, YubiKey, AWS KMS, and software vaults. Each backend implements the same interface. The agent never knows which backend holds its keys.

Leased Secrets – in Autonomy and Ockam

  • Identity-bound token leasing. The agent proves its identity over a secure channel. A lease issuer exchanges that proof for a short-lived, scoped token.
    Each LeaseToken is bound to the requesting agent’s Ockam Identifier. The lease issuer reads the authenticated identity from the secure channel, not from a claim. The long-lived secret never leaves the lease issuer.
  • Automatic lifecycle. Tokens refresh at half-TTL and revoke on expiry. Each agent gets its own scoped token.
    A background processor manages the lease lifecycle: refresh before expiry (at half-TTL, minimum 5-second interval), revoke when the lease expires or the agent’s identity is revoked. Every lease is per-agent: the billing agent gets its own Stripe token, the support agent gets a different one scoped to refunds. No shared credentials, full attribution.
  • Alternative: gateway injection. A proxy intercepts outgoing requests and injects credentials at the transport layer. The agent never touches any secret at all.
    Ockam and Autonomy also support a gateway pattern: an HTTP interceptor sits between the agent and the downstream service, injecting the leased token into Authorization headers before the request leaves the node. The agent’s code makes a normal HTTP call; the credential appears only in transit, never in agent memory. The trade-off: the gateway must understand every protocol and request format passing through it, but the agent holds zero secrets.

Credentials – What can this agent do, and on whose behalf?

Identity says who. A credential says what, and on whose behalf.

Two kinds exist: bearer credentials like API tokens, which are secrets that must be brokered; and cryptographic credentials, signed statements verifiable without calling the issuer.

An authenticated identity tells us who. An authenticated credential tells us what, and on whose behalf. Bearer credentials are secrets: API tokens, database passwords, OAuth access tokens. Agents need them because existing systems run on them, but the agent should never see the raw secret. We need infrastructure to broker short-lived tokens scoped to a specific agent and its needs. Cryptographic credentials are different: an authority attests that a specific identity has certain attributes. Each credential is bound to a subject’s cryptographic identifier and carries attributes as key-value pairs: role, scope, delegation context, expiration. The signature proves authenticity without calling the issuer.

A credential should capture who delegated, for what purpose, for how long. Each delegation should only narrow authority, never amplify it. An agent should present its own identity; it should never impersonate the delegator.

An agent’s identity tells us that it is requesting an action. A cryptographic credential tells us whether it is authorized to do it, and by whom. Not just this agent should have access to order history but this agent should have access because a specific customer delegated that authority to resolve this complaint. When an agent delegates to another agent, the credential captures who delegated to whom, for what purpose, and for how long. Each delegation can only reduce authority, never amplify it. A credential verifier sees three distinct facts: which agent is acting, who delegated authority to it, and what constraints apply.

Credentials – in Autonomy and Ockam

  • Short-lived, scoped, signed. Each credential carries typed attributes as key-value pairs, not opaque scopes. Bound to the subject’s Identifier. Default TTL: 30 minutes.
    Each credential carries created_at and expires_at (default TTL: 30 minutes). Attributes are typed key-value pairs, not opaque scopes. The credential is signed with the issuer’s CredentialSigning purpose key and bound to the subject’s Identifier. An OAuth access token says “the authorization server approved these scopes at some point.” An Ockam credential says “this specific authority attests that this specific identity has these specific attributes, valid until this timestamp.”
  • Any identity can issue; trust is scoped. Issuance is permissionless. Whether an attestation is trusted depends on Trust Contexts: which issuer is authoritative for which attributes.
    Any Ockam identity can sign a credential about any other identity. But whether that attestation is trusted depends on Trust Contexts: a policy that specifies which issuer is authoritative for which attributes. Issuance is permissionless. Trust evaluation is not. This is what enables delegation: any agent can issue a credential to a sub-agent, and the verifier decides whether to trust it.
  • Delegation as credential chains. Each delegation narrows authority. The agent never impersonates the delegator.
    When an agent delegates to a sub-agent, it issues a credential. The sub-agent presents its own identity with the delegation credential. Each delegation can only reduce authority, never amplify it. The agent never impersonates the delegator. A verifier sees three facts: which agent is acting, who delegated authority, and what constraints apply.
  • Offline verification. The verifier checks the signature chain locally: issuer’s ChangeHistory, PurposeKeyAttestation, credential signature. Thousands of checks per second, no central bottleneck.
    Credential verification requires no call to the issuer. The verifier checks the signature chain locally: issuer’s ChangeHistory, PurposeKeyAttestation linking the signing key to the issuer, and the credential signature itself. Thousands of credential checks per second without a central authorization service in the path. Enrollment – in Autonomy and Ockam

  • One-time enrollment token. A provisioner asks the Authority to issue a single-use token with attributes. The ticket bundles the code with everything the new agent needs to find and authenticate the Authority.
    The provisioner is a human or automated system that decides which attributes a new agent should receive. The Authority creates a single-use code, stores it with the requested attributes, and returns it. The enrollment ticket bundles this code with everything the new node needs to find and authenticate the Authority: project route, authority identity, and authority route.
  • Local identity creation. The new node creates its own Ed25519 identity in milliseconds. No CA, no DNS, no operator in the loop.
    Identity creation is entirely local. The node generates an Ed25519 key pair and computes its Identifier as the SHA256 hash of the first change. No certificate authority, no DNS registration, no human approval needed. In Autonomy, every agent creates its own cryptographic identity this way before enrolling.
  • Secure channel + present token. The node opens a secure channel to the Authority and presents the one-time code. The Authority verifies, consumes the token, and adds the node as a member.
    The new node opens a mutually authenticated secure channel to the Authority and presents the one-time code. The Authority verifies the code is valid and unused, consumes it (so it cannot be replayed), and adds the node’s Identifier as a project member with the pre-defined attributes. The enrollment token is single-use by design: one token, one identity, one enrollment.
  • Credentials issued. The Authority issues credentials with the enrolled attributes. The agent is now a project member.
    Once enrolled, the Authority issues credentials carrying the attributes specified in the original token. These credentials are short-lived (default 30-minute TTL) and automatically refreshed. The enrolled identity can now present these credentials over secure channels to access project resources.

Secure Channels – How do agents talk safely?

Agents act on what they receive. Compromised messages mean compromised actions. The communication channels to agents must guarantee message authenticity, integrity, confidentiality, and forward secrecy.

Agents operate across clusters, clouds, and organizations. They are vulnerable to injection, and unlike traditional software, they take autonomous actions. A manipulated agent doesn’t just leak data. It destructs. Agents handle financial records, medical data, customer PII. That data has value to attackers long after the session ends. If an attacker records encrypted traffic and later compromises an agent’s channel key, can they decrypt the recording? Forward secrecy guarantees they cannot. A compromised channel key does not unlock past messages.

TLS authenticates domains, not agents. Bearer tokens identify accounts, not agents. Neither side knows who it is actually talking to. Agents must authenticate each other’s identifiers, attributes, and delegation context.

A TLS certificate authenticates a domain, not an agent. A thousand agents sharing one domain all look the same to a client. On the client end, bearer tokens tell the receiving agent which account a request belongs to, not which of the thousand agents sent it. Today’s most prominent agent communication protocols build on these same primitives: TLS to authenticate the server, OAuth to authorize the client. They authenticate services and accounts, not the actual agents. After the handshake, each agent must be confident of the other’s identifier, attributes, and delegation context, because it has verified signatures only that agent’s private key could produce.

Relays, gateways, and load balancers see plaintext when TLS terminates at the boundary. End-to-end encryption runs between the two end agents. Intermediaries forward ciphertext they cannot decrypt.

Most agents aren’t a public service. They communicate across a few specific systems and organizations. To reach each other, agents route through relays, gateways, and load balancers. TLS terminates at the gateway. The gateway sees plaintext. It may read it, modify it, log it, or leak it. End-to-end encryption solves this: the channel runs between the two end agents, guaranteeing integrity and confidentiality regardless of how many hops the transport takes. Without end-to-end encryption, every intermediary in the path is an attack surface that can expose sensitive data and manipulate highly capable agents. In Autonomy, any agent can create a mutually authenticated, end-to-end encrypted channel with any other agent. Autonomy’s encrypted relay service routes these channels across private network boundaries.

Routing – in Autonomy and Ockam

  • onward_route / return_route. Each message carries a route. Each hop removes itself from onward_route, adds itself to return_route. The reply travels back without the sender knowing the full path.
    Ockam messages carry two ordered lists of addresses: onward_route and return_route. The sender places the full path in onward_route and its own address in return_route. Each intermediate hop pops its address from the front of onward_route and pushes it onto the front of return_route. The replier reverses: it makes the incoming return_route its outgoing onward_route. This means any worker can reply without knowing the topology.
  • Transport-agnostic, multi-hop. Routes span TCP, WebSocket, UDP, Bluetooth. Any number of intermediaries. Each hop forwards without reading the payload.
    Addresses in the route refer to workers (named processing units), not network endpoints. A route can span multiple transports: the first hop might be TCP, the second WebSocket, the third a relay. The routing layer does not care. The routing protocol handles any number of hops with the same algorithm. Each intermediary applies one rule: pop itself from onward_route, push itself onto return_route, forward. The intermediary never needs to inspect the payload. This is the foundation that makes end-to-end encrypted secure channels possible: intermediaries route ciphertext they cannot decrypt.

Multi-Hop Routing – in Autonomy and Ockam

  • Sender decoupled from topology. The sender specifies a route, not connections. Add a relay, move a service, switch transports. The application does not change.
    The sender constructs a route as an ordered list of addresses. It does not open connections, manage sockets, or track which hop is where. The transport layer handles connections independently. The routing algorithm is uniform across all intermediaries: each hop removes its own address from onward_route and adds it to return_route, then forwards. Same code at every hop, regardless of position. Network topology can change without modifying any application code. The route is the only thing that changes.

Secure Channels – in Autonomy and Ockam

  • Between identities, not network addresses. The channel runs between two Ockam Identities. It works across TCP, WebSocket, relay. Topology does not matter.
    The secure channel is between two Ockam Identities, not between two IP addresses or domains. Two agents in different clouds, different clusters, different organizations create a secure channel without exposing either to the public internet. The relay forwards ciphertext it cannot decrypt.
  • Noise XX: mutual auth in three messages. Both sides exchange and verify identities, purpose key attestations, and credentials during the handshake.
    After message 3, each agent knows the other’s identifier, attributes, and delegation context, because it has verified signatures only that agent’s private key could produce. The diagram below shows the full protocol: Key Establishment Phase (three messages), then Encrypted Transport Phase with derived session keys k1 and k2.
  • No cipher negotiation. Compile-time choice (Noise_XX_25519_AESGCM_SHA256). No downgrade attacks. No fallback to weaker ciphers.
    Cipher suite is a compile-time choice, not a runtime negotiation. All participants use the same primitives. There is no “fallback to weaker cipher” path for an attacker to exploit.
  • Forward secrecy with rekeying. Ephemeral keys per session. Symmetric key rotated every 32 messages. Trail of Bits audited.

Portals – in Autonomy and Ockam

  • TCP Portal = Inlet + Outlet. An Inlet listens on a local TCP port. An Outlet connects to a target TCP service. Between them: a secure channel through Ockam’s relay.
    A Portal is a pair of components that tunnel a TCP connection through an Ockam secure channel. The Inlet accepts connections from local TCP clients and forwards their data through the encrypted channel. The Outlet receives the data and connects to the target TCP service. The entire path between Inlet and Outlet is end-to-end encrypted, mutually authenticated, and routed through Ockam’s relay infrastructure.
  • Transparent to applications. Existing TCP clients and servers work unchanged. The portal handles encryption, routing, and identity verification.
    Applications connect to a local TCP port (the Inlet) as if the service were running locally. On the other end, the target service receives a standard TCP connection from the Outlet. Neither the client nor the server knows about the secure channel in between. No code changes, no SDK integration, no certificate management. The portal is invisible infrastructure.
  • Private services, no public exposure. The target service stays in a private network. No public IP, no open firewall port. The secure channel reaches it through the relay.
    The Outlet runs in the same private network as the target service. It creates an outbound connection to the relay. No inbound ports need to be opened. No public IP is required. The service remains invisible to the internet while being reachable through the authenticated, encrypted portal. This is how agents in Autonomy reach private databases, APIs, and services across organizational boundaries.

Access Control – Can this agent do that?

Agents choose tools autonomously. Every invocation must be evaluated: allow or deny. Access control must sit outside the agent, in infrastructure the agent cannot bypass.

Agents act by invoking tools. A tool call can run a function, call an API, query a DB, write a file, run a shell command, send a message to another agent, or delegate work to a sub-agent. System instructions guide behavior, but can’t enforce it. Inputs like user messages and tool call results routinely include untrusted content. A prompt injection in a tool result can override system instructions. Every tool invocation must be evaluated before it executes: allow or deny.

Access Controls – in Autonomy and Ockam

  • Cryptographically authenticated attributes. A single verified chain from authority to access decision.
    Policies are evaluated against attributes that were attested by a trusted authority and delivered through a verified credential chain. Not self-asserted claims. Not role memberships in an external directory. The identity, the credential, the secure channel, and the policy form a single verified chain from authority to access decision.
  • Three gates, all closed by default. Tool availability, invocation, external service. No policy = deny.
    Tool availability: credential attributes determine which tools an agent can reach. Invocation: before a tool call executes, infrastructure evaluates whether this specific invocation is permitted. External service: the agent proves its identity to a broker and receives scoped credentials the service validates independently. If no policy exists, deny.
  • ABAC policy language. S-expression boolean expressions over credential attribute names. Dynamic updates.
    Policies are S-expression boolean expressions over attribute names. Operators: and, or, not, =, !=, member?, exists?. A boolean shorthand lets you write simple checks without S-expressions. Dynamic policies can be updated while workers are running.
  • Every invocation, not once at session start. Revoke mid-task; the next call is denied.
    Credentials are checked on every tool call, not once at connection time. Revoke authority mid-task, and the next invocation is denied. No stale permissions. Thousands of agents invoke tools simultaneously; enforcement is distributed, not centralized.

Policies – in Autonomy and Ockam

  • Boolean expressions over attributes. Policies are S-expression boolean expressions. Operators: and, or, not, =, !=, member?, exists?. Each expression evaluates to true or false against the sender’s credential attributes.
    Each attribute name/value pair from a credential is added to the environment as subject.attribute_name. The policy expression is recursively evaluated against this environment. The result is a boolean: allow or deny. Values can be String, Int, Float, Bool, or Seq (a sequence of values). The member? operator checks if a value is present in a Seq.
  • Boolean shorthand. For simple attribute checks, write deploy_agent and staging_access instead of the full S-expression. Readable for operators, precise for machines.
    The boolean shorthand translates attribute names to (= subject.name "true"). You can use infix and, or, not operators and string comparisons like component="web". Parentheses group expressions with precedence rules: not > and > or. An Ockam Identifier can be used directly as a policy, translating to (= subject.identifier "I84502ce...").
  • Dynamic, distributed. Policies can be updated while workers are running. Manage centrally, distribute asynchronously. Enforcement is local, not centralized.
    PolicyAccessControl models a Resource/Action pair. Policies for that resource and action can be modified even if the worker they are attached to is already started. Thousands of agents invoke tools simultaneously. A central decision point in the path of every invocation cannot keep up. Distribute policies to every enforcement point. Evaluate locally.

Traces – Who did what, and why?

An agent invokes a tool, interprets the result, decides what to do next. Traces capture every step: each tool invocation, each decision, each result. They are the audit trail and the debugging tool.

Traces must capture every step at every level of detail: each tool invocation, each decision, each intermediate result. They are how you debug failures, analyze errors, and build evaluations. They are also the audit trail: who did what, under whose authority. No agent is reliable on day one. Traces are how agents improve.

Not just “agent X did Y” but who authorized it, each intermediate agent, and the constraints each delegation imposed. The full delegation chain.

A record that says “agent X performed action Y” tells you what happened. It does not tell you who authorized it. The trace must capture the full delegation chain: the human who delegated, each intermediate agent, and the constraints each delegation imposed.

A log entry can be edited or fabricated. A record signed with the agent’s private key cannot be forged or repudiated. No agent is reliable on day one. Signed, attributable traces are how agents improve.

The signature ties the action to the agent that performed it and the delegation chain that authorized it. When an agent hits a case it cannot handle, it escalates to a human. The trace captures how the exception was resolved: what the human considered, why they chose a particular path. That reasoning, previously locked in someone’s head, is now captured. Engineers study the traces, build evaluation sets, and iterate on the agent until it handles similar cases on its own. The next escalation doesn’t happen.
